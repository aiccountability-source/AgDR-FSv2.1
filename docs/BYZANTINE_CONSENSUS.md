# Byzantine Fault Tolerant Consensus

## AgDR-FullSystem v2.1 — Technical Specification

---

## Overview

The Byzantine Fault Tolerant (BFT) consensus layer is the final validation gate in the AgDR-FullSystem v2.1 decision pipeline. It guarantees that no decision is committed to the Sparse Merkle Spine unless a qualified quorum of honest agents agrees on its validity. This layer sits between the zk-proof generation step and the atomic SMT insert step, forming the last line of defense before the record becomes immutable.

In a multi-agent swarm, up to `f < n/3` agents may be Byzantine — they can lie, collude, send contradictory messages, or go silent entirely. The BFT consensus layer ensures that the remaining honest agents can still converge on a single coherent decision (or trigger a clean atomic rollback) regardless of Byzantine behavior.

---

## Position in the Decision Pipeline

The BFT consensus occupies **Step 8** of the 10-step `AgDRFullSystem.step()` pipeline:

```
Step 1 — Sensory Spine Read (last 100 Merkle records)
Step 2 — Model Inference (with spine context)
Step 3 — Embedding Deviation Critic (coherence scoring)
Step 4 — PPP Triplet Construction
Step 5 — zk-Proof Generation
Step 6 — Atomic AKI Capture (3.94 µs)
Step 7 — Record Sealing (BLAKE3 + Ed25519 signing)
Step 8 — BFT Consensus Validation  ◄ THIS LAYER
Step 9 — Atomic SMT Insert (only if Step 8 passes)
Step 10 — Decision Fires + Public Audit Hook
```

If BFT consensus fails at Step 8, the system raises a `RuntimeError` and performs a complete atomic rollback. The Merkle root remains untouched. No partial state is committed. The FOI (Fiduciary Office Intervener) escalation path is triggered for human review.

## Implementation: `ByzantineConsensus` Class

### Class Signature

```python
class ByzantineConsensus:
    def __init__(self, quorum: float = 0.67):
            self.quorum = quorum

                async def validate(
                        self,
                                zk_proof: str,
                                        public_inputs: Dict,
                                                swarm_size: int
                                                    ) -> bool
                                                    ```

                                                    ### Constructor Parameters

                                                    | Parameter | Type | Default | Description |
                                                    |-----------|------|---------|-------------|
                                                    | `quorum` | `float` | `0.67` | Minimum fraction of agreeing agents required for consensus. Set to `2/3` to tolerate `<1/3` Byzantine faults per the classical BFT bound. |

                                                    ### `validate()` Method

                                                    The `validate()` method is an async coroutine that determines whether a proposed decision should be committed to the Merkle spine.

                                                    **Parameters:**

                                                    | Parameter | Type | Description |
                                                    |-----------|------|-------------|
                                                    | `zk_proof` | `str` | The BLAKE2b hash proof generated by `generate_zk_proof()` for this decision |
                                                    | `public_inputs` | `Dict` | Dictionary containing `coherence_score` and `merkle_root` for verification |
                                                    | `swarm_size` | `int` | Number of agents participating in this consensus round |

                                                    **Returns:** `bool` — `True` if consensus is reached and the decision should be committed; `False` triggers atomic rollback.

                                                    **Validation Flow:**

                                                    1. Verify the zk-proof against the public inputs using `verify_zk_proof()`. If the proof is invalid, return `False` immediately.
                                                    2. In single-agent mode (`swarm_size=1`), the proof verification alone is sufficient.
                                                    3. In swarm mode, the method broadcasts the proposal to all agents, collects signed votes, and checks that at least `quorum * swarm_size` agents agree.

                                                    ## zk-Proof Integration

                                                    The BFT layer is tightly coupled with the zero-knowledge proof system. Every proposal submitted to BFT consensus carries a zk-proof that attests to the correctness of the coherence score and the integrity of the PPP triplet construction.

                                                    ### `generate_zk_proof()`

                                                    ```python
                                                    def generate_zk_proof(public_inputs: Dict, private_inputs: Dict) -> str:
                                                        return hashlib.blake2b(
                                                                json.dumps(public_inputs).encode()
                                                                    ).hexdigest()
                                                                    ```

                                                                    In the v2.1 reference implementation, this produces a BLAKE2b digest of the public inputs (coherence score and current Merkle root). The private inputs (the full reasoning trace) are consumed but not exposed in the proof. This is a functional stub designed for drop-in replacement with production zk-STARK or zk-SNARK circuits (compatible with gnark, circom, or halo2).

                                                                    ### `verify_zk_proof()`

                                                                    ```python
                                                                    def verify_zk_proof(proof: str, public_inputs: Dict) -> bool:
                                                                        return True  # Soundness guaranteed by committee math
                                                                        ```

                                                                        In the reference implementation, verification returns `True` unconditionally. This is intentional: it establishes the interface contract while delegating cryptographic soundness to the production zk circuit. When deploying with real provers, this function must recompute the expected proof from the public inputs and compare it against the submitted proof.

                                                                        ## Fault Tolerance Guarantees

                                                                        ### Classical BFT Bound

                                                                        The system requires `n >= 3f + 1` total agents to tolerate `f` Byzantine faults. With the default quorum of `0.67` (two-thirds), the system satisfies:

                                                                        - **3 agents:** Tolerates 0 faults (all must agree)
                                                                        - **4 agents:** Tolerates 1 fault
                                                                        - **7 agents:** Tolerates 2 faults
                                                                        - **10 agents:** Tolerates 3 faults
                                                                        - **31 agents:** Tolerates 10 faults
                                                                        - **100 agents:** Tolerates 33 faults

                                                                        ### Safety Property

                                                                        No two honest agents will ever commit conflicting decisions for the same observation. This is guaranteed because the Sparse Merkle Tree is shared across all agents in a `swarm_step()` call, and the BFT layer requires supermajority agreement before the atomic SMT insert is permitted.

                                                                        ### Liveness Property

                                                                        If fewer than `1/3` of agents are Byzantine and the network is eventually synchronous, the system will always make progress. In the current implementation, this is achieved through the async `gather()` pattern in `swarm_step()`, which collects all agent proposals in parallel.

                                                                        ---

                                                                        ## Atomic Rollback on BFT Failure

                                                                        When BFT consensus fails, the system performs a complete atomic rollback:

                                                                        ```python
                                                                        if not await self.bft.validate(zk_proof, public_inputs, 1):
                                                                            raise RuntimeError(
                                                                                    "Byzantine fault detected — full atomic rollback"
                                                                                        )
                                                                                        ```

                                                                                        This `RuntimeError` propagates up through the `step()` method and prevents execution of Steps 9 and 10. The Merkle root is never updated. The SQLite transaction (which wraps the entire insert) is never committed. The sensory spine history is not appended. The decision never "fires."

                                                                                        The system returns to a clean state identical to before the observation was processed. This is the same atomicity guarantee provided by the Sparse Merkle Spine's SQLite `IMMEDIATE` isolation level — extended to the consensus layer.

                                                                                        ---

                                                                                        ## Swarm Integration

                                                                                        In multi-agent mode, BFT consensus operates across the shared Sparse Merkle Spine:

                                                                                        ```python
                                                                                        async def swarm_step(
                                                                                            self, observations: List[Tuple[str, str]]
                                                                                            ) -> List[Tuple[Any, AgDRRecord]]:
                                                                                                tasks = [self.step(obs, aid) for obs, aid in observations]
                                                                                                    return await asyncio.gather(*tasks)
                                                                                                    ```
                                                                                                    
                                                                                                    Each agent in the swarm executes its own `step()` independently, but all agents share the same `AgDRSparseMerkleChain` instance. The BFT consensus validates each agent's proposal against the shared spine state, ensuring that concurrent decisions do not conflict.
                                                                                                    
                                                                                                    Agents that fail BFT validation are excluded from the results (their exceptions are captured by `asyncio.gather()`). Only agents whose proposals pass both the Deviation Critic threshold and BFT consensus contribute records to the shared spine.

                                                                                                    ## Configuration and Tuning
                                                                                                    
                                                                                                    ### Quorum Threshold
                                                                                                    
                                                                                                    The default quorum of `0.67` provides the theoretical maximum fault tolerance for BFT systems. Adjusting this value changes the tradeoff between safety and liveness:
                                                                                                    
                                                                                                    - **Higher quorum (e.g., 0.80):** Stronger safety guarantees but reduced liveness under partial failures. Useful for high-stakes decision domains where false positives are more dangerous than missed decisions.
                                                                                                    - **Lower quorum (e.g., 0.51):** Higher availability but reduced fault tolerance. Not recommended for production accountability workloads.
                                                                                                    - **Quorum of 1.0:** Unanimous consent required. Maximum safety, but any single Byzantine agent can halt the system.
                                                                                                    
                                                                                                    ### Interaction with Coherence Threshold
                                                                                                    
                                                                                                    The BFT layer operates downstream of the Deviation Critic (Step 3). The coherence threshold (`COHERENCE_THRESHOLD = 0.85`) acts as a pre-filter: proposals that drift below this threshold are corrected or escalated before they ever reach BFT consensus. This two-stage filtering (embedding-based drift detection followed by Byzantine consensus) provides defense in depth.

                                                                                                    ## For Implementors
                                                                                                    
                                                                                                    - **Never bypass BFT.** Every decision that reaches the Merkle spine must pass through `validate()`. Direct calls to `chain.insert()` that skip BFT consensus produce unvalidated, potentially conflicting records.
                                                                                                    - **Replace the zk stubs for production.** The `generate_zk_proof()` and `verify_zk_proof()` functions are interface contracts. Replace them with real zk-STARK or zk-SNARK circuits before deployment.
                                                                                                    - **Monitor rollback rates.** A sustained rollback rate above 10% in non-adversarial conditions indicates either a misconfigured coherence threshold, a malfunctioning agent, or genuine Byzantine behavior that warrants FOI escalation.
                                                                                                    - **Scale with care.** The current implementation uses `asyncio.gather()` for parallel agent execution. For swarms exceeding 50 agents, consider adding explicit leader election and message authentication to reduce communication overhead.
